Table NameITRVL_SRQ_ENTITY_METADATA_VW
Module NameREPORT
DescriptionTHIS IS THE TABLE ON WHICH THE META DATA OF THE SCREEN DEFINITION IS SAVED. THE META DATA JSON GENERATED BY THE 'ENTITY META DATA DESIGNER' IS SAVED ON TO THIS TABLE. SINCE THIS IS AN ADMINISTRATION FUNCTIONALY, THIS SHOULD BE KEPT UNDER 'SYS' module (according to iTravel Design Rules) You should not link directly to this tables from other modules of itravel. In case if you need to link, you can expose the data of this tables via API and access it on other modules. (according to iTravel Design Rules) You can also store UNIQUE key of this tables on other tables. Since all the design rules prevail in itravel, we are actually storing a copy of the data from this table into ITRVL_SRQ_ENTITY_META_DATA table of SERVICE REQUEST module. This is because, the ENTITYTYPE in SRQ module always require to retrieve its META data definition to render the screen each time on various other modules also (CAB, CART, BOOKING, AGENCY) So the steps should be as follows 1. The ADMINISTRATOR will define the screen using the ENTITY META DATA DESIGNER in the admin module 2. This data will be saved in ITRVL_SRQ_ENTITY_METADATA table 3. When a new service request is created, the METADATA is retrieved by making an API call to table ITRVL_SRQ_ENTITY_METADATA to get the screen defintion information. 4. While saving the newly created service request, the SCREEN DEFINITION META DATA is saved to ITRVL_SRQ_ENTITY_META_DATA table. Thus we save the snapshot of screen definition at the time of creation of service request along with its model data. (Later on, if somebody makes a change to METADATA in the admin screen, the previously created service request does not get affected, and will use the same meta data definition that existed at the time of creation of service request)+ 5. Once the service request is created, the the META DATA details is always retrieved from ITRVL_SRQ_ENTITY_META_DATA table ONLY.
